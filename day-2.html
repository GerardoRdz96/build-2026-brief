Source: Microsoft Developer (official live stream) Focus: GitHub Copilot · VS Code · Foundry Format: live notes, lightly edited

The big picture

Why Day 2 mattered
THE THROUGHLINE

The whole industry is converging on the same primitive: the agent skill.

A skill is a small, reusable unit — a name, a description, and instructions an agent loads on demand — that gives an agent new capabilities and works the same way across tools. Anthropic introduced the format; at Build, Google's Addy Osmani (author of the open-source agent-skills project and a contributor to Google's Agent Development Kit) presented the same model on Microsoft's stage. A portable skill standard across Claude Code, GitHub Copilot, and Google's ADK is a big deal: the workflows you build become reusable across every assistant, not locked to one vendor.

What shipped

The announcements that matter
01

GitHub Copilot app

An "agent-native desktop experience" — not an editor, not a CLI. It works like a session manager: run several agents in parallel, each isolated in its own git worktree, with the code intentionally in the background so you stay focused on outcomes. A grounded note from the panel: the realistic number of parallel agents is around three to four, not the "50 agents" you see on social media.

GA 1.0
02

GitHub Copilot SDK

Packages the agent runtime behind the Copilot CLI and app — the prompts, tools, and context loop — so you can embed it into any application. Available in six languages (Node, TypeScript, Python, Go, .NET, and now Rust and Java). The team is rewriting the runtime in Rust for performance. It already powers products like Office, Excel, and more inside Microsoft.

ROLLING OUT
03

MAI Code Flash

Microsoft's first coding model built from scratch (not a fine-tune of someone else's). It's a fast, efficient model purpose-built for the Copilot workflow — understanding repository context, using tools, and making targeted edits — and it was trained with reinforcement learning against the real product so that training, evaluation, and production share one environment. Rolling out in the VS Code model picker.

04

Copilot "Auto" model selection

Picks the best available model for your task automatically, in real time, with zero configuration — across the IDE, the CLI, and the cloud. It takes the "which model should I even use, and what about the one that ships tomorrow?" decision off your plate.

UPDATED
05

GitHub Copilot CLI

Built-in custom agents (explore, plan, code review) that compose with skills; parallel sessions you can switch between; an experimental "autopilot" loop; and a voice mode that runs on local models (including English and Spanish), so no cloud round-trip. Installable via Homebrew and WinGet with auto-update.

06

TypeScript 7.0 ("TSGo")

The TypeScript compiler has been ported from JavaScript to Go for native speed and multi-core parallelism — roughly a 10x speedup. On the VS Code codebase (about 1.5 million lines), a full build dropped from ~1m38s to ~10s. Already used in production at companies like Sentry, Slack, Notion, and Figma.

Patterns

What good agentic engineering looks like now
01

It's a spectrum

Vibe coding sits at one end, disciplined spec-driven work at the other. The more diligence you apply, the further toward "agentic engineering" you are. Pick the point that matches the stakes — there's no single right answer.

02

Skills follow the lifecycle

The most useful skills map to define → plan → build → verify → review → ship. The same discipline teams already trust, now packaged so an agent can follow it.

03

Review with a different model

The CLI's review command runs multiple models (e.g. Sonnet, GPT, Gemini) and surfaces only the issues they agree on. Different training data catches different mistakes — don't let the model that wrote the code be the only one that checks it.

04

The CLI is back

"Your MCP server should probably just be a skill and a CLI." Agents are extremely good at using command-line tools, and skills load progressively, which keeps the context window lean. The honest caveat: authentication is where MCP still earns its place.

Going deeper

Two engineering segments worth your time

🧰 Models are not interchangeable

  • A "harness" is the system around the model — it edits files, runs the terminal, manages context, and loops. The model only returns tokens.
  • Every model has its own "personality," so the harness is re-tuned per model. Even within a family, behavior shifts; across vendors it shifts more, and they call tools differently.
  • The four things teams tune for each new model: system prompt, available tools, context management, and the agent loop.
  • Practical migration tip: when a new model lands, start from the previous model's prompt and iterate from there.

📊 How to actually evaluate a model

  • Treat public benchmarks as regression tests, not quality scores — modern models saturate them.
  • Build your own internal benchmark of real tasks; run each model several times to average out non-determinism.
  • Score with a "resolution rate": did the run hit a checklist of assertions for that task? Yes → resolved.
  • Offline evals were called one of the highest-leverage tools for anyone building on top of these models. Worth digging into.

Lines worth keeping

Quotes from the day
"Build your own assistant that works how you work." Forget installing something that magically works out of the box. Build the workflow that fits you, and automate the simple things you do every day. That was called the most transformative step.
"The tool you install matters less than the workflow you build around it." Agent skills, and the various skill packs, are all useful — but the real work is assembling the workflow that fits your project, and it isn't the same across projects.
"Cognitive debt / cognitive surrender." The quiet risk of always deferring to whatever the model recommends. Stay in the driver's seat.
Read the trajectory, not every line. A useful rule of thumb: for a throwaway project, you don't need to read the code; for something you'll maintain for months, understand the architecture and read the core files.
